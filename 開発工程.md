# **PI-ZZA 開発工程・技術仕様書**

本ドキュメントは、プロジェクト PI-ZZA の完遂に必要な技術的要件、モジュール構成、およびテスト計画を定義する。

## **0. 設計原則（Phase 0 Foundation）**

- **言語方針**:
  - オーケストレータ: **Go 1.22+**
  - フォーク元 OSS は**元言語のまま**保持
    - M1 Seed: Go
    - M2 Kitchen: TypeScript (Firecrawl) — REST 越境
    - M3 Delivery: Python (browser-use)
    - M4 Box: Python (Streamlit + SQLite)
- **通信方式**: モジュール間は **gRPC** ([buf](https://buf.build) で proto 管理)
- **開発方法論**: **厳格 TDD** (Red → Green → Refactor)
- **コミット規約**: **Conventional Commits** + release-please 自動化
- **モノレポ構造**: polyglot monorepo (`api/` `cmd/` `internal/` `services/` `third_party/`)
- **公開形態**: GitHub Public OSS (`github.com/clearclown/pizza`, MIT)

詳細は [ARCHITECTURE.md](./ARCHITECTURE.md) / [docs/architecture.md](./docs/architecture.md) / [CONTRIBUTING.md](./CONTRIBUTING.md) 参照。

## **1. モジュール構成 (Modular Architecture)**

プロジェクトは、3 つの外部 OSS をフォーク・統合し、独自のオーケストレーター (Go) で制御する。

### **M1: Seed Module (生地生成 / Dough)**

- **実装言語**: Go
- **フォーク元**: [gosom/google-maps-scraper](https://github.com/gosom/google-maps-scraper) (MIT) + [googlemaps/google-maps-services-go](https://github.com/googlemaps/google-maps-services-go) (Apache-2.0)
- **gRPC サービス**: `pizza.v1.SeedService`
- **機能**:
  - 指定ポリゴン範囲をグリッド分割し、検索漏れをゼロにする (`internal/grid/`)
  - 店舗名、住所、PlaceID、公式 URL を収集
- **技術的課題**: API コストの最適化と、重複データの自動マージ

### **M2: Kitchen Module (下ごしらえ / Toppings)**

- **実装言語**: TypeScript (フォーク元のまま)
- **フォーク元**: [mendableai/firecrawl](https://github.com/mendableai/firecrawl) (**AGPL-3.0**)
- **ライセンス分離**: **REST 越境のみ**。Go バイナリにリンクしない（[docs/license-compliance.md](./docs/license-compliance.md)）
- **gRPC サービス**: 将来 `pizza.v1.KitchenService`（現在は直接 REST）
- **機能**:
  - 収集した URL を巡回し、HTML をクリーンな Markdown へ変換
  - 「会社概要」「店舗一覧」「採用情報」などの重要セクションを優先抽出
- **技術的課題**: スクレイピング対策（Anti-bot）の回避

### **M3: Delivery Module (AI 判定エージェント / Courier)**

- **実装言語**: Python 3.11+
- **フォーク元**: [browser-use/browser-use](https://github.com/browser-use/browser-use) (MIT)
- **gRPC サービス**: `pizza.v1.DeliveryService`
- **LLM プロバイダ**: Anthropic / OpenAI / Gemini **全切替対応** (registry パターン)
- **機能**:
  - Markdown 情報から判断できない場合、AI がブラウザを起動
  - 「店舗検索画面」で条件を変えて検索し、直営店数を特定する等の自律操作
  - 最終的な「フラグ判定（単一/複数加盟、20 店舗以上か）」のロジック実行
- **技術的課題**: LLM プロンプトエンジニアリング、プロバイダ別の出力差吸収

### **M4: Box Module (BI・可視化)**

- **実装言語**: Python (Streamlit + SQLite)
- **gRPC サービス**: `pizza.v1.BIService`
- **機能**:
  - 収集したメガジーリストを地図上にマッピング
  - 1 店舗あたりの推定収益を算出し、スコアリングする

## **2. 開発フェーズ (Development Roadmap)**

| フェーズ | 期間 | 内容 | 成果物 |
| :---- | :---- | :---- | :---- |
| **Phase 0: Foundation** | 1 週間 | OSS 公開準備・TDD 基盤・proto 契約・CI。本プロジェクトは Phase 0 から着手。| GitHub public repo + scaffolding |
| **Phase 1: Grid & Scrape** | 1 週間 | Google Maps からの網羅的な店舗抽出エンジン (M1 Seed) の完成 | 座標ベース抽出器 |
| **Phase 2: Markdownize** | 1 週間 | Firecrawl 統合 + コンテンツ抽出器 (M2 Kitchen) | Markdown パイプライン |
| **Phase 3: AI Brain** | 2 週間 | browser-use のプロンプトエンジニアリングと判定精度の向上 (M3 Delivery) | 判定エージェント |
| **Phase 4: BI Interface** | 1 週間 | Streamlit による低コストなデータ閲覧・出力機能の実装 (M4 Box) | PI-ZZA Dashboard |

## **3. テスト項目 (Testing Protocol)**

すべてのテストは **TDD Red フェーズ** で先に書き、実装（Green）で通す。

### **3.1 単体テスト (Unit Testing)**

- [ ] **Grid Test** (`internal/grid/grid_test.go`): 緯度経度計算において、指定範囲が 100% カバーされているか（**Phase 0 で Red 済**）
- [ ] **Parser Test** (`internal/toppings/parser_test.go`): Firecrawl が主要な FC サイトのフッターから「会社概要 URL」を抽出できるか
- [ ] **Sanitization Test** (`internal/slice/dedupe_test.go`): 取得した CSV から、不正な文字列や重複レコードが排除されているか
- [ ] **Scoring Test** (`internal/scoring/mega_test.go`): 20 店舗閾値でのメガジー判定（**Phase 0 で Red 済**）
- [ ] **LLM Provider Test** (`services/delivery/tests/test_providers/`): 各プロバイダが統一 interface を満たすか（**Phase 0 で Red 済**）

### **3.2 連結テスト (Integration Testing)**

- [ ] **E2E Flow** (`test/e2e/pipeline_test.go`): ブランド名入力から、最終的なメガジー特定（CSV 出力）までがエラーなく完走するか。testcontainers-go で docker compose 起動
- [ ] **Pipeline Test** (`internal/oven/pipeline_test.go`): bufconn で gRPC スタブを立て、Seed → Kitchen → Delivery の順序制御が正しいか（**Phase 0 で Red 済**）
- [ ] **API Backoff Test** (`internal/oven/retry_test.go`): GCP API や LLM API のレート制限がかかった際、正しく指数バックオフ（Retry）が行われるか

### **3.3 精度ベンチマーク (Accuracy Benchmark)**

- [ ] **Classification Accuracy**: 人間が判定した「直営・FC」の正解データ 100 件に対し、AI の判定一致率が 90% を超えているか
- [ ] **Recall Rate**: 指定エリア内の実店舗数に対し、抽出された店舗データが 95% 以上を網羅しているか

## **4. 拡張性への要求 (Scalability)**

- **汎用リサーチ機能**: `cmd/pizza` に `--mode custom` を追加し、FC 以外（例: 特定の設備を持つ不動産、特定のメニューがある飲食店）の検索にも対応させる
- **マルチ AI 対応**: OpenAI, Gemini, Claude を切り替え可能にし、コストと精度のバランスを調整できるようにする（**Phase 0 から実装済**）
- **Kitchen プロバイダ切替**: Firecrawl 以外の Markdown 変換器（Crawl4AI, Jina Reader）を `KITCHEN_PROVIDER` 環境変数で選択可能に（将来）

## **5. 品質保証 (QA Guardrails)**

- CI 必須: `go test -race -cover` / `pytest` / `buf lint` / `buf breaking` / `golangci-lint` / `ruff`
- ブランチ保護: main への直 push 禁止、1 approver + green CI 必須
- 自動脆弱性スキャン: CodeQL (Go / Python)、Dependabot
- リリース自動化: release-please (Conventional Commits → CHANGELOG + tag)
- upstream 同期: 週次 cron で `git subtree pull` → PR 自動作成
